# PM 测试平台（pm-test）

> 最后更新：2026-06-04
> 配套：`docs/agents.md`（Agent 索引）、`.claude/rules/`（规则文件）

本文档是测试平台的**唯一职责文档**，合并了"职责范围"和"上游交付物验收清单"。从测试视角定义：平台做什么、测什么、上游要交什么、责任如何划分。

---

## 0. 平台定位

**我们不是开发，不是构建，不是运维。我们是测试。**

```
[上游开发] ──交付产物──> [测试平台] ──测试报告──> [下游运维/发布]
                              │
                              ├─ 构建测试（验证产物可跑）
                              ├─ 业务测试（验证业务正确）
                              └─ 失败打回/上报
```

核心立场：
- **验证上游的产物能不能跑**（构建测试）
- **验证跑起来的应用业务对不对**（业务测试）
- **发现上游问题就如实打回，不替上游修**

## 1. 上游必须交付的清单

### 1.1 交付物清单（11 大类）

| 类别 | 必交项 | 形式 | 不交的后果 |
|------|--------|------|----------|
| **代码制品** | 编译后的可运行产物（dist/、build/、target/、jar、binary 等） | tar.gz/zip 归档 + manifest（含 SHA256） | 打回：无法部署 |
| **依赖锁定** | 依赖清单 + 锁文件（package.json + lock 文件 / requirements.txt / go.mod / pom.xml 等） | 与制品同包 | 打回：无法保证依赖一致 |
| **数据库** | SQL 文件（全量 dump / 增量 migrate / seed 脚本） | 单独的 SQL 文件，明确标注：① 全量 or 增量 ② 基线版本 ③ 执行顺序 | 打回：无法判断初始化方式 |
| **配置模板** | 环境变量清单 + .env.example / application.yml 模板 | 与制品同包 | 打回：.env 字段缺失 |
| **版本说明** | 本次发布的 CHANGELOG：新增/修改/删除的功能、影响范围 | markdown 或 txt | 打回：无法评估测试范围 |
| **部署文档** | 启动命令、监听端口、依赖的中间件（DB/Redis/Nginx 等）及其版本 | 与制品同包 | 打回：无法启动 |
| **测试凭据** | 测试账号（test_ 前缀）、API Key、Token | 单独交付（脱敏或专用） | 打回：无凭据无法验证登录 |
| **种子数据** | 测试用的基础数据（部门、角色、字典等） | SQL 或 JSON | 平台自行创建 |
| **API 文档** | OpenAPI/Swagger 或接口清单（路径、方法、参数、返回） | 与制品同包或 URL | 平台无法设计 API 测试 |
| **构建说明** | 构建命令、目标产物路径、构建环境要求 | README 或部署文档 | 打回：无法验证产物 |
| **单元测试覆盖** | L1 单元测试报告（覆盖率 ≥ 60%，目标 80%） | lcov / cobertura / jacoco 等 | 打回：未跑单元测试 / 覆盖率不达标 |
| **关键模块单测清单** | 核心业务模块的单元测试文件列表 + 通过率 | 文件清单 | 平台无法验证单测充分性 |

### 1.2 上游必须明确标注的"覆盖 vs 增量"

| 场景 | 上游必须说明 |
|------|-------------|
| 数据库文件 | 每次都全量替换？基于上一版本增量？SQL 文件的执行顺序？ |
| 配置文件 | .env 哪些字段是新增？哪些是修改？哪些是废弃？ |
| 接口变更 | 新增哪些接口？废弃哪些接口？哪些接口签名变了？ |
| 依赖变更 | 哪些依赖升版？哪些降级？哪些删除？是否影响 API 兼容性？ |

**没说清 = 上游没交完 = 打回。**

## 2. 测试平台的职责（详细）

### 2.1 阶段 1：构建测试（Build Verification）

**目标**：验证上游交付物**能不能正常部署、启动、运行**。

#### 2.1.1 制品完整性验证

| 检查项 | 方法 | 失败处理 |
|--------|------|----------|
| 归档文件存在 | `tar -tzf` 列出 | 打回：制品丢失 |
| 归档未损坏 | `tar -tzf` 无错误 | 打回：制品损坏 |
| manifest 校验 | 对比 SHA256 | 打回：产物不一致 |
| 关键文件齐全 | 检查 dist/、package.json、schema 等 | 打回：关键文件缺失 |
| 无敏感信息泄露 | 检查无 .env（真实值）、无密钥文件 | 打回：违反安全规范 |

#### 2.1.2 依赖解析验证

| 检查项 | 方法 | 失败处理 |
|--------|------|----------|
| 锁文件存在 | 检查 lock 文件 | 打回：缺 lock 文件 |
| 锁文件与制品匹配 | lock 文件中的版本与制品声明一致 | 打回：lock 与制品不一致 |
| 依赖能成功安装 | `npm install` / `pnpm install` / `pip install` 等 | 打回：依赖装不上 |
| 关键依赖就位 | `@nestjs/common`、`prisma` 等关键包 require 成功 | 打回：关键依赖缺失 |
| 引擎版本匹配 | Node.js/Java/Python 版本符合 package.json 声明 | 打回：运行时版本不对 |

#### 2.1.3 数据库初始化验证

| 检查项 | 方法 | 失败处理 |
|--------|------|----------|
| 数据库能连接 | 用上游提供的连接信息连接 | 打回：连接信息错 |
| SQL 文件能执行 | 按上游顺序执行无语法错误 | 打回：SQL 语法错 |
| 表结构完整 | 关键表存在、字段完整 | 打回：表结构不完整 |
| 字符集正确 | utf8mb4，无乱码 | 打回：字符集不对 |
| 种子数据就位 | 关键种子数据存在 | 打回：种子数据缺失 |
| 数据量符合预期 | 关键表记录数 > 0（如果有 seed） | 打回：数据导入失败 |

#### 2.1.4 配置完整性验证

| 检查项 | 方法 | 失败处理 |
|--------|------|----------|
| .env 包含所有必需字段 | 字段数 = 上游 .env.example 字段数 | 打回：字段缺失 |
| 关键密钥存在 | JWT_SECRET、API Key 等非空 | 打回：密钥缺失 |
| 数据库连接串正确 | 能连接到目标 DB | 打回：连接串错 |
| 端口不冲突 | 启动前检查端口空闲 | 平台问题：换端口 |
| 中间件地址正确 | Redis/MQ 等地址能 ping 通 | 打回：中间件地址错 |

#### 2.1.5 服务启动验证

| 检查项 | 方法 | 失败处理 |
|--------|------|----------|
| 后端进程能启动 | 启动后进程存在 | 打回：启动失败 |
| 端口在监听 | `ss -tlnp` / `netstat` | 打回：服务挂了 |
| 无模块解析错误 | 日志中无 `Cannot find module` | 打回：依赖问题 |
| 无环境变量缺失错误 | 日志中无 `xxx is required` | 打回：配置问题 |
| 无数据库连接错误 | 日志中无 DB connection refused | 打回：DB 配错 |
| 启动时间合理 | < 60s | 打回：启动慢 |

#### 2.1.6 健康检查验证

| 检查项 | 方法 | 失败处理 |
|--------|------|----------|
| healthCheck 端点 200 | curl | 打回：服务假死 |
| 响应时间合理 | < 2s | 打回：性能问题 |
| 响应内容正确 | 包含预期字段 | 打回：接口异常 |

#### 2.1.7 页面渲染验证

| 检查项 | 方法 | 失败处理 |
|--------|------|----------|
| HTML 200 | curl 首页 | 打回：前端起不来 |
| 页面有实际内容 | `browser_snapshot` 看到非空白 | 打回：dist 缺包/路径错 |
| 控制台无 JS 错误 | `browser_console_messages` 无 error | 打回：模块解析失败 |
| 无 404 资源 | 静态资源全 200 | 打回：dist 不完整 |

#### 2.1.8 登录验证

| 检查项 | 方法 | 失败处理 |
|--------|------|----------|
| 登录接口 200/401 | curl `/api/auth/login` | 打回：API 不通 |
| 用上游凭据能登录 | 实际登录成功 | 打回：密码哈希格式错 |
| 登录后跳正确页 | 跳 `postLoginPath` | 打回：登录逻辑错 |
| 用户信息正确 | 看到期望的用户名/角色 | 打回：用户状态错 |

### 2.2 阶段 2：业务测试（Business Testing）

**目标**：验证应用**业务逻辑**的正确性。

#### 2.2.1 测试层级

| 层级 | 谁负责 | 工具 | 覆盖范围 | 占比 |
|------|--------|------|---------|------|
| **L1 单元** | **上游开发** | 项目自带 / Vitest / pytest / JUnit | 单个函数/类逻辑（mock 外部） | 60% 最低 / 80% 目标 |
| **L2 API** | **测试平台** | Vitest + supertest / Jest + supertest | 接口契约、参数校验、返回结构、错误码 | 40% 最低 / 70% 目标 |
| **L3 E2E** | **测试平台** | Playwright | 完整业务流程，用户视角 | 核心流程 100% / 关键路径 100% |
| **L4 UI** | **测试平台** | Playwright | 表单校验、响应式、视觉、交互 | 主页面 100% / 核心交互 |

**L1 单元测试是上游开发的职责，不是平台的事。** 平台不替开发写单元测试、不重跑开发代码层级的单测、不修改开发编写的单测代码。

平台对 L1 的处理方式（仅验收）：
- 上游交付时**必须**提交：单元测试覆盖率报告（lcov/cobertura/json 等格式）
- 覆盖率不达标 → **打回上游**让开发补
- 关键模块的单元测试文件清单 → 平台可抽样 review
- 平台**不**写 L1 单元测试，**不**为 L1 失败负责

平台对 L2/L3/L4 的处理方式（自主测试）：
- 平台基于上游交付的制品，**自主设计** L2 API 测试用例
- 平台基于上游交付的应用，**自主设计** L3 E2E 用例（用户视角）
- 平台基于上游交付的前端，**自主设计** L4 UI 用例
- 这些用例的代码、运行、维护**全部由平台负责**

#### 2.2.2 L3 E2E（主要测试方式）

- **用户视角**：模拟真实用户操作（点按钮、填表单、看结果）
- **核心流程必测**：登录、CRUD、关键业务链路
- **不直接调 API**：除登录等特殊场景
- **不直接改 DB**：用 UI 触发业务

#### 2.2.3 L2 API（辅助交叉验证）

**用途**：当 E2E 失败时，定位问题在前端还是后端。

```
E2E 失败
  ├─ 调用对应 API → 失败 → 后端 Bug
  └─ 调用对应 API → 通过 → 前端 Bug
```

#### 2.2.4 测试范围确定

测试范围由三个输入决定（按优先级）：

1. **`case/` 目录用户案例**（最高优先）—— 上游或产品提供的业务场景
2. **`reports/` 变更报告**（次优先）—— 上游 CHANGELOG 影响到的功能
3. **自主探索**（兜底）—— 平台基于应用探索生成

### 2.3 阶段 3：测试报告

**必须生成**（无论通过或失败）：

| 文件 | 内容 |
|------|------|
| `results/{module}/progress.txt` | TC 级 PASS/FAIL/SKIP 状态 |
| `results/{module}/report.md` | 模块详细报告（结果概览、缺陷、修复记录、截图） |
| `results/summary.md` | 跨模块汇总（通过率） |

### 2.4 阶段 4：交付下游

**交付条件**（必须全部满足）：

- ✅ 所有模块 L3 E2E 全部 PASS
- ✅ 所有失败用例有明确归因（应用 Bug / 平台问题）
- ✅ 修复的应用 Bug 已重测通过
- ✅ `summary.md` 报告完整
- ✅ **用户确认交付**

**未达条件 → 不交付**，出失败报告给上游或上报。

## 3. 失败归因矩阵（强制）

构建测试和业务测试失败时，**必须明确归因**，不能含糊：

| 失败类型 | 责任归属 | 处理 | 谁做 |
|----------|---------|------|------|
| 制品缺失/损坏 | **上游** | 打回 + 报告 | 主会话 |
| 锁文件缺失/冲突 | **上游** | 打回 + 报告 | 主会话 |
| SQL 语法错/缺文件 | **上游** | 打回 + 报告 | 主会话 |
| 覆盖/增量未标注 | **上游** | 打回 + 报告 | 主会话 |
| .env 字段缺失 | **上游** | 打回 + 报告 | 主会话 |
| 启动命令错 | **上游** | 打回 + 报告 | 主会话 |
| 密码哈希错/用户状态错 | **上游** | 打回 + 报告 | 主会话 |
| 部署脚本/平台环境问题 | **平台** | 修复 + 记录 | build-verifier |
| 端口冲突（平台侧） | **平台** | 报告用户 | 主会话 |
| **业务测试失败** | **上游（应用 Bug）** | 修复 + 上报 | healer |
| E2E 失败 + API 通过 | **上游（前端 Bug）** | 上报 | healer |
| E2E 失败 + API 失败 | **上游（后端 Bug）** | 上报 | healer |
| 测试用例本身错 | **平台** | healer 修复 | healer |

**核心原则**：

> 平台不修上游的问题。**修上游的产物 = 帮上游掩盖问题 = 下次还会犯。**

## 4. 平台边界（强制）

### 不做

- ❌ 跑 `pnpm build` / `npm run build` / `mvn package`（这是上游的事）
- ❌ 改上游代码
- ❌ 改上游配置（.env.example、application.yml）
- ❌ 改 SQL 文件
- ❌ 修上游的密码哈希
- ❌ 跳过构建测试直接做业务测试
- ❌ 把应用 Bug 标 PASS 蒙混过关
- ❌ 自动交付下游

### 做

- ✅ 验证上游产物能跑
- ✅ 验证跑起来的应用业务对
- ✅ 发现问题就如实打回
- ✅ 维护当前版本测试状态
- ✅ 全部通过后交付下游

## 5. 测试策略分层决策

| 场景 | 用什么测 | 原因 |
|------|---------|------|
| 上游交付了一个新功能 | L3 E2E | 验证用户能正常使用 |
| 上游改了接口 | L2 API 优先 + L3 E2E | API 验证契约，E2E 验证联通 |
| 上游改了前端组件 | L4 UI 优先 + L3 E2E | UI 验证交互，E2E 验证流程 |
| 上游修了 Bug | L3 E2E 重跑原失败用例 | 回归验证 |
| 上游改了数据库 schema | L2 API 验证新字段 | 验证数据契约 |
| 上游升级了依赖版本 | L3 E2E 跑核心流程 | 验证无回归 |
| 上游交付未提供 L1 单测 | **打回上游** | L1 是开发责任，不在平台范围 |

## 6. 跨阶段硬性检查

每个阶段必须等上一个阶段**全部通过**才能启动：

| 阶段 | 必须验证 |
|------|----------|
| 上游交付 → 构建测试 | 制品完整、SQL 文件齐全、.env 完整、覆盖/增量标注清晰 |
| 构建测试 → 业务测试 | 健康检查 PASS + 页面渲染 PASS + 登录验证 PASS |
| 业务测试 → 出报告 | 测试运行完成（有 PASS 也有 FAIL 都算） |
| 报告 → 下游交付 | 全部模块通过 + **用户确认交付** |

**任一不满足 → 不进入下一阶段。**

## 7. Agent 职责矩阵

### 7.1 主会话

| 必须做 | 明确不做 |
|--------|---------|
| 接收上游交付物，启动构建测试 | 改上游产物 |
| 调度三个阶段：构建测试 → 业务测试 → 报告 | 跳过构建测试 |
| 构建测试失败 → 出打回报告 | 静默放过构建问题 |
| 业务测试失败 → 决定修复策略（healer）| 改源码 |
| 所有通过后主动询问是否交付下游 | 自动交付 |
| **状态文件管理**：调用 `updateStage()` 推进 pipeline-state | 替 Agent 改产物 |

### 7.2 build-verifier（构建测试 Agent）

| 必须做 | 明确不做 |
|--------|---------|
| 验证产物完整性、依赖解析、数据库初始化、配置完整性 | 改上游产物 |
| 实际启动服务 + 健康检查 + 页面渲染 + 登录验证 | 跑业务测试 |
| 出具环境验证报告（results/build/） | 改测试代码 |
| 失败 → 出具打回报告（具体到责任归属） | 含糊报"部署失败" |
| 写 `environment.json.validator.*` 段 | 改 `analyzer.*` 段 |

### 7.3 playwright-test-planner

| 必须做 | 明确不做 |
|--------|---------|
| 读 `case/` 用户案例（最高优先） | 写测试代码 |
| 读 `reports/` 变更报告（次优先） | 跳过 case/ 自主探索 |
| 启动时 `browser_navigate` 目标 URL | 假设浏览器处于任何特定页面 |
| 每个需要认证的操作自行完成登录 | 跳过用户确认直接进入 Generate |
| **生成含 UI Map 的模块计划**（generator 用来直接生成模式） | 写代码 |
| 等待用户多轮确认（可增删 TC、调整步骤、调整优先级） | 一次性给完不等用户审阅 |

### 7.4 playwright-test-generator

| 必须做 | 明确不做 |
|--------|---------|
| **生成模式自检**：先看计划文件有没有 UI Map → 有走直接生成、无走录制 | 一律用录制模式 |
| 直接生成模式：从 UI Map 提取选择器，**不执行浏览器** | 录制后再"重构"选择器 |
| 录制模式：每个 TC 独立录制会话，操作→读 log→写文件 | 一个录制会话做多个 TC |
| 写 `tests/{level}/{module}/tc-{编号}-{简称}.spec.ts` | 写到项目根目录 `tests/` 或 `e2e/` |
| 截图路径**必须**含 `test_project/<NN-Project>/` 完整前缀 | 用相对路径 |
| 等待策略优先 Playwright 智能等待 | `waitForLoadState('networkidle')` |
| 测试数据用 `test_` 前缀 | 删/改已有非测试数据 |
| 断言必须与计划预期完全一致 | 写"自适应"断言 |

### 7.5 playwright-test-healer

| 必须做 | 明确不做 |
|--------|---------|
| `test_run` → `test_debug` → 系统化诊断 | 凭猜测修 |
| 修复范围：`tests/`、`results/`、`playwright.config.ts`、`environment.json` | 改 build/dev/ 产物、删其他模块结果 |
| 每个 TC 最多 3 次修复尝试 | 无限制尝试 |

| 更新 `progress.txt`、`report.md`、`summary.md` | 跨模块引用截图 |

## 9. 实际教训（来自 01-oa-llm 部署案例）

之前我们混淆了"构建项目"和"验证构建"两个角色：

| 原做法 | 问题 | 修正 |
|--------|------|------|
| 我们自己跑 `pnpm build` | 越权做了上游的事 | 应基于上游提供的 dist/ 验证 |
| 我们自己 `pnpm install` | 上游 lock 文件有问题，应该打回 | 应先检查 lock 文件 |
| 我们自己改 SQL 导入策略 | 上游没说清是覆盖还是增量 | 应先看上游的覆盖/增量标注 |
| 我们修数据库密码 | 上游提供的密码哈希不对，应打回 | 应打回让上游修 |
| 我们修 `.env` 字段 | 上游 `.env.development` 不完整，应打回 | 应打回让上游补全 |

**核心原则**：

> 平台不修上游的问题，平台只发现并打回上游的问题。
> 修上游的产物 = 帮上游掩盖问题 = 下次还会犯。

构建测试的**真正价值**是发现问题、强制上游修，而不是替上游擦屁股。

---

## 10. 关键不变量（强制）

1. **构建测试不通过 → 必须打回**，不允许"修一下就好了"
2. **业务测试不通过 → 必须如实上报**，不允许降级断言标准
3. **所有通过 + 用户确认 → 才能交付下游**，不允许自动交付
4. **平台 Agent 不改上游产物**，发现问题是报告，不是修
5. **覆盖 vs 增量必须明确**，上游没说清就当上游问题打回
6. **失败归因必须明确**，不允许含糊报"测试失败"
7. **E2E 失败时用 API 交叉验证定位问题归属**
